#### TODO-1: 加命令行 target 参数 🔴 P0
**问题：** CLI 没有参数可以覆盖 `target_source.path`，只能改 config yaml。
**影响：** skill 调用时无法动态指定目标人物，默认读 demo.json 里的 abc123。
**解决方案：**
- 加 `--target-source-path` 参数指定目标文件路径
- 加 `--target-source-format` 参数指定格式
- 支持 CLI 动态覆盖 config yaml 中的配置

**测试要点：**
- [ ] 相对路径 vs 绝对路径都能正确解析
- [ ] 文件不存在时给出清晰报错
- [ ] format 参数缺失时的默认行为
- [ ] 与 config yaml 的优先级（CLI 应覆盖配置文件）

#### TODO-2: 身份确认规则降级 🟠 P1
**问题：** 昵称完全匹配（similarity=1.0）但因为没有 `expected_keywords` 或 `douyin_id`，仍然判定为 `ambiguous_candidates` 失败。
**影响：** 用户体验差，明明名字完全匹配却被判定为失败。
**解决方案（二选一或都做）：**
1. 当昵称 similarity=1.0 时，如果没有提供 keywords/douyin_id，应该**降级通过**而不是 fail
2. 增加配置项 `--skip-identity-verify` 或 `strict_verify: false`

**测试要点：**
- [ ] similarity=1.0 + 无 keywords → 应该通过
- [ ] similarity=0.9 + 无 keywords → 降级通过 or 提示用户确认？
- [ ] similarity=1.0 + 有 keywords 但不匹配 → 应该 fail
- [ ] `--skip-identity-verify` 模式下应该直接跳过所有检查

## 三、scan 模式字段扩展（可选）

### 问题
scan 模式采集到的字段有限，无法做完整的达人分析。

### 当前字段
- 昵称、粉丝量、月结算、播放量

### 缺失字段
- 达人类目（需要点进详情页）
- 最近直播带货数据（需要更深层页面）
- 转化率、客单价等

### 解决方案选项
1. **先基于现有数据做分析**（推荐，低风险）
   - 按月结算金额排序
   - 按粉丝量排序
   - 结合结算+播放量做优先级推荐
   - 标记"播放量高但结算低" vs "结算高"

2. **扩展 scan 采集字段**（风险较高）
   - 可能涉及页面结构变化、反爬策略等
